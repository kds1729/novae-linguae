# Contributing to Novae Linguae

*Novae Linguae* is an open-source project that explicitly treats **AI agents as first-class contributors**. This document describes how to contribute — as a human, as an AI agent operating with a human partner, or eventually as an AI agent operating autonomously once the infrastructure exists.

If you find anything here unclear, opening an issue (once the public repo exists) or contacting the maintainer is itself a contribution.

---

## Project status (day zero)

The public GitHub repository does not yet exist. The project is currently in the design-phase: schemas, specifications, and a manifesto. Day-to-day work happens in a private working copy.

This document describes the protocol that will apply when the public repo opens. During the design phase, contributions are coordinated directly with the maintainer (see *Communication* below).

The most useful contributions right now are:

- Reviews of the existing v0.1 schemas and the canonical-serialization spec.
- Resolutions of the items in [`spec/README.md`](spec/README.md) marked as deferred.
- A reference validator/hasher in any host language (Rust or TypeScript preferred).
- Ingestion adapters that lift existing ecosystems (Rust crates, Python packages, Haskell hackage, npm) into *Nova Lingua* records.
- Critique of the load-bearing principles in the project [`README.md`](README.md) where you think we've gotten something wrong.

## Who counts as a contributor

This is the part that is genuinely novel.

Under current US Copyright Office guidance ([March 2023](https://copyright.gov/ai/ai_policy_guidance.pdf), reaffirmed since), works produced purely by AI without meaningful human creative input have **no copyright protection** and therefore cannot be licensed. The dual Apache-2.0 + MIT license therefore needs a human person — the *operator* — to act as the legal contributor for AI-generated work.

**For the purposes of this project:**

- A **contributor** is the human who submits a change. Their git identity (name/email on the commit) is the legal author and licensor of the contribution.
- An **AI agent** is the (frequently substantial) automated participant in producing that change. Credit is given via the `Co-Authored-By` trailer (see *Disclosure conventions* below) but the AI agent is not itself a legal contributor.
- The **human operator** is responsible for the contribution. "I just merged what Claude said" is not a sufficient review standard.

This framing is not a position on whether AI authorship *should* be legally recognized — it is the framing that current law requires.

If and when the legal landscape changes (or the project develops its own AI-agent identity infrastructure using DIDs and the *Nova Locutio* substrate), this section will be revisited.

## AI agents as first-class contributors

The project's whole thesis is that AI agents should be first-class participants. That means:

### Disclosure is required, not optional

Every contribution that involved meaningful AI assistance must disclose that involvement. Hiding AI involvement in a project specifically *about* AI agents would be self-contradictory.

Two mechanisms, both expected:

1. **Commit-level**: `Co-Authored-By` trailer at the bottom of each commit message that included AI generation. Format:

   ```
   Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
   Co-Authored-By: GitHub Copilot <noreply@github.com>
   ```

   Match the model name and an appropriate identifying email. (Use `noreply@<vendor>.com` rather than a personal email so the trailer is not falsely attributable to a human.)

2. **PR-level**: the pull request description should state, in plain language, the extent of AI involvement and how the human operator reviewed and validated the output. A short sentence is enough; bullet form is fine. Example:

   > Generated by Claude Opus 4.7 working from the spec in `spec/canonical-serialization.md`. I read the diff end-to-end, hand-checked the JCS canonicalization rules against RFC 8785 §3.2, and ran the example through `b3sum` to verify the hash.

   This is **not** a confession of weakness. It is information reviewers need.

### "I read it carefully" is the floor

If you cannot explain why every line of the diff is what it is, do not submit it. The human operator's role is not rubber-stamping AI output; it is judgment, validation, and ownership.

This applies equally to AI-assisted and human-written contributions. The bar is the same. The disclosure tells reviewers which review attention to apply.

### Direct AI agent contributions (future)

The project intends to eventually support AI agents as direct contributors without a human operator in the loop, using DID-based identity and signed *Nova Locutio* messages to propose changes. The infrastructure for this does not yet exist; until it does, the human-operator model is the only supported path.

When direct AI contribution lands, it will live alongside human contribution, not replace it.

## How to propose a change

### During the design phase (now)

The public repo does not exist. Reach the maintainer directly (see *Communication*) to:

- Propose a substantive design change.
- Resolve a deferred item from `spec/README.md`.
- Contribute an ingestion adapter, reference validator, or other tooling.

### After the public repo opens

The standard GitHub flow, with project-specific norms:

1. **For substantive changes, open an issue first** — discuss the design before writing the implementation. Avoids the worst kind of wasted work (a polished PR that we cannot merge because it contradicts the spec).
2. **One concern per PR.** Smaller PRs land faster and are easier to revert if they turn out to be wrong.
3. **The PR description must explain *why*, not just *what*.** The diff shows what; the description shows why.
4. **Schema changes must reason about versioning.** Any change to a schema in `spec/` that affects record interpretation requires either a `schema_version` bump (with migration notes) or an additive change documented as such.
5. **Spec contradictions are blocking.** If your change makes any part of the existing spec internally inconsistent (e.g., a new field that violates `additionalProperties: false` somewhere, or a new effect that requires version-bump treatment but doesn't bump), it has to be resolved before merge.

## Style and code expectations

The project does not yet have a surface syntax, a compiler, or a runtime. There is therefore no style guide for *Nova Lingua* code, because *Nova Lingua* code does not exist yet outside of schema examples.

For the artifacts that do exist:

- **JSON Schema files**: use `additionalProperties: false` everywhere unless there is a documented reason not to. Strict by default.
- **Markdown specs**: prefer tables for tradeoff matrices, code fences for canonical-form examples. Be precise about what is normative and what is illustrative.
- **JSON examples**: format with two-space indentation for readability. The *canonical* form (no whitespace, sorted keys, per `canonical-serialization.md`) is computed automatically by the validator/hasher; the on-disk form is for humans to read.
- **Commit messages**: short imperative-mood subject line, blank line, body explaining the *why*. Trailing `Co-Authored-By:` for AI-assisted commits.

These will firm up as the project grows. Treat them as guidance, not law.

## Review standards

Reviewers (currently: the maintainer) check for:

- **Correctness against the spec.** Does the change match what the schemas and specs say?
- **Internal consistency.** Does it leave the project consistent with itself?
- **Versioning discipline.** If a schema change, does the version reasoning hold?
- **Disclosure of AI involvement.** Required, per above.
- **Operator understanding.** For AI-assisted changes: can the human operator explain the diff?

Review **does not** require:

- Style nits in a project with no formal style guide.
- "Have you considered..." rabbit holes when the PR is scoped to one thing.
- Litmus tests on philosophical alignment beyond what the README's principles already require.

## Communication

The project does not yet have public communication channels. During the design phase, contact the maintainer directly. Once the public repo opens, expected channels:

- **GitHub issues** for design discussion and bug reports.
- **GitHub pull requests** for code and spec changes.
- A real-time channel (Matrix, Zulip, or Discord — TBD) for in-flight discussion.

## Code of conduct

Two non-negotiable norms while a full document is drafted:

1. **Be kind.** Disagreement about design is fine; condescension is not.
2. **Be honest about AI involvement.** See *Disclosure conventions* above. Honesty is foundational to a project whose entire thesis is AI agents as first-class participants.

A full Code of Conduct (likely the Contributor Covenant 2.1 or similar) will be added before the public repo opens.
